## 1. 引言

### 1.1 编写目的

本文档为"通用流处理任务管理平台"的概要设计说明书，旨在说明系统的技术选型、整体架构、模块划分以及核心数据流。文档面向开发团队、评审专家以及后续维护人员，作为详细设计和编码实现的依据。

### 1.2 项目背景

在建设流计算系统时，经常遇到流式数据的加工处理需求，如流量接入、数据字段补数、数据格式转换、数据多路转发等。这些需求在多系统间存在大量重复建设问题。为提升科技部门对流式数据加工处理的开发效率，需要研发一个专门面向流处理场景的通用流处理任务管理平台。

本平台允许用户通过可视化拖拉拽的方式编排数据处理作业（DAG），支持 CSV、Excel、MySQL、Kafka 等多种数据源的输入输出，以及 XML↔JSON 转换、字段拼接、Redis 扩充等数据治理操作。

### 1.3 术语说明

| 术语 | 说明 |
|------|------|
| DAG | 有向无环图 (Directed Acyclic Graph)，作业中控件的执行顺序图 |
| 控件 (Component) | 数据处理的原子单元，分为输入控件、处理控件、输出控件三类 |
| 作业 (Job) | 由多个控件通过连线组成的完整数据处理流水线 |
| React Flow | 基于 React 的可视化节点编辑器库，用于实现拖拉拽画布 |
| FastAPI | Python 异步 Web 框架，本项目后端基础 |
| Polars | Rust 实现的高性能 DataFrame 库，用于 CSV 等数据处理 |
| pybind11 | C++ 与 Python 的绑定工具，用于关键路径性能加速 |
| TopologicalSorter | Python 标准库提供的拓扑排序器，用于 DAG 执行序求解 |
| Kahn 算法 | 经典的 DAG 环路检测算法，基于入度队列 |

### 2.2 选型理由

#### 2.2.1 为什么选 Python FastAPI 作为统一后端

本项目的后端需求分为两部分：REST API 层（用户管理、控件管理、作业 CRUD）和流执行引擎层（DAG 解析、控件执行）。两种方案对比如下：

| 维度 | Java Spring Boot + Python 引擎 | Python FastAPI 统一后端 (选定) |
|------|-------------------------------|------------------------------|
| 运行时进程 | 2 个（JVM + Python） | 1 个（Uvicorn） |
| 内存占用 | 2.5~3 GB (JVM 1.5G + Python 1G) | ~400 MB |
| 跨语言通信 | REST + Redis 队列，延迟高 | 同一进程直接调用，无开销 |
| 开发效率 | CRUD 需 4 文件 (entity/mapper/service/controller) | CRUD 单文件函数 |
| 热重载 | Java 10-20s | Python 2s |
| 团队学习成本 | 需要 Java + Python 两套技能 | 全组 Python 即可 |

**结论：** Python FastAPI 统一后端在开发效率、运行时开销、部署复杂度三个维度均优于混合架构。8VC 32G 测试环境下，Java 方案基础开销达 2.5GB，Python 方案仅 400MB，省出的内存可直接用于数据处理。

#### 2.2.2 为什么选 React Flow

React Flow 是目前最成熟的 React 节点编辑器库：

1. **自定义节点** — 支持完全自定义的节点渲染，本项目据此实现了输入/处理/输出三类控件的差异化视觉样式（接口显隐、颜色区分、状态指示灯）
2. **连线管理** — 内置 Handle 机制，支持自定义连接验证规则
3. **拖拽支持** — 原生支持从外部 DOM 元素拖拽创建节点
4. **插件系统** — MiniMap、Controls、Background 等开箱即用
5. **社区生态** — 对比 Vue Flow，React Flow 的文档、示例、issue 数量均大幅领先

#### 2.2.3 为什么选 Polars 而非 Pandas

Polars 是基于 Rust 实现的 DataFrame 库，通过 Python 接口调用。在处理 CSV 文件读写、字段拼接等操作时，Polars 比 Pandas 快 5~30 倍，且内存占用更低。对于 8VC 32G 机械硬盘环境下的性能测试至关重要。

#### 2.2.4 为什么需要 C++ pybind11 加速

针对字段拼接（A+B→C）和大文件 CSV 分块读取两个 CPU 密集型操作，编写了 C++ 加速模块。在大数据量（>100 万行）场景下，C++ 版本比纯 Python 实现快 5~10 倍。pybind11 可以将 C++ 代码编译为 Python 可直接 import 的 .pyd 模块，无缝集成到引擎中。

## 4. 系统架构设计

### 4.1 总体架构

```
┌─────────────────────────────────────────────────────────┐
│                    前端 (React 18)                        │
│  Ant Design | React Flow | Zustand | React Router       │
│              :3000 (Vite Dev Server)                     │
├─────────────────────────────────────────────────────────┤
│                REST API (JSON)                            │
├─────────────────────────────────────────────────────────┤
│                后端 (Python FastAPI) :8080                │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────────────────┐    │
│  │  api/    │  │ engine/  │  │ components/        │    │
│  │ 用户CRUD │  │ DAG执行器 │  │ 12个内置控件       │    │
│  │ 控件CRUD │  │ 拓扑排序 │  │ 4输入+4处理+4输出  │    │
│  │ 作业CRUD │  │ 依赖传播 │  │ @register_component│   │
│  │ 上下线   │  │ 状态追踪 │  │                    │    │
│  └──────────┘  └──────────┘  └────────────────────┘    │
│                      │  (C++ pybind11 .pyd 加速)        │
├─────────────────────────────────────────────────────────┤
│           SQLite (开发)  /  MySQL 8 (生产)               │
└─────────────────────────────────────────────────────────┘
```

### 4.2 架构分层说明

#### 4.2.1 表示层 (Presentation Layer)

React 18 单页应用 (SPA)，通过 Axios 发送 JSON 请求到后端 API。核心页面包括仪表盘、用户管理、控件管理、作业列表和作业编辑器。作业编辑器采用 React Flow 实现可视化拖拉拽编排 DAG。

#### 4.2.2 业务逻辑层 (Business Logic Layer)

FastAPI 路由层负责处理 HTTP 请求，包含用户/控件/作业的 CRUD 操作。作业上线/下线操作通过调用引擎层的 JobRunner 来控制作业的生命周期。所有业务数据持久化到 MySQL。

#### 4.2.3 执行引擎层 (Execution Engine Layer)

DAG 执行器 (DAGExecutor) 是引擎的核心，负责解析 JSON 格式的 DAG 配置，使用 Python 标准库的 TopologicalSorter 进行拓扑排序，然后按序执行各控件。支持上游节点失败后的自动依赖传播（跳过下游受影响节点）、节点执行状态追踪以及执行日志收集。

#### 4.2.4 控件层 (Component Layer)

采用"装饰器注册"的插件化架构。每个控件实现继承自 BaseComponent 抽象基类，通过 @register_component 装饰器注册到全局注册表。引擎通过控件名称查找并执行。新增控件只需继承基类 + 加装饰器 + 在 main.py 中 import，无需修改引擎代码。

#### 4.2.5 数据层 (Data Layer)

使用 SQLAlchemy 2.0 异步引擎，开发环境默认 SQLite 免安装零配置启动，生产环境通过环境变量切换为 MySQL 8.x。所有 ORM 模型统一继承自 DeclarativeBase。

## 6. 应用数据流图

### 6.1 用户创建作业流程

```
用户 ──→ 前端 (React)
          │
          ├─ 1. 从左侧控件面板拖拽控件到画布
          ├─ 2. 连线连接控件 → 形成 DAG
          ├─ 3. 点击控件 → 右侧配置面板填写参数
          ├─ 4. 点击 [保存] → POST /api/jobs
          │      └─ dag_config JSON 存入 MySQL
          └─ 5. 点击 [上线] → POST /api/jobs/{id}/online
                 └─ 引擎开始执行
```

### 6.2 作业上线执行流程

```
POST /api/jobs/{id}/online
          │
          ▼
FastAPI: 更新 job.status = "online"
          │
          ▼
JobRunner.start_job(job_id)
          │
          ▼
asyncio.create_task(_run_loop)
          │
          ▼
┌── 循环执行 ──────────────────────────┐
│                                       │
│  1. 从 MySQL 读取 dag_config          │
│  2. DAGExecutor.execute(config)       │
│       ├─ TopologicalSorter 拓扑排序   │
│       ├─ 按序执行各控件               │
│       ├─ 上游失败 → 跳过下游          │
│       └─ 写 Execution 记录            │
│  3. 等待 5s → 下一轮                  │
│                                       │
└───────────────────────────────────────┘
          │
          ▼  (用户点下线)
POST /api/jobs/{id}/offline
→ stop_flags[job_id] = True → 循环退出
```

### 6.3 数据流处理流程（CSV 输入 → 字段拼接 → MySQL 输出）

```
[CSV 输入控件]            config: {path: "data.csv"}
     │                    Polars: pl.read_csv(path)
     │                    output: {data, columns, rows}
     ▼
[字段拼接控件]            config: {field_a:"A", field_b:"B", field_c:"C"}
     │                    读取上游 data, 拼接 A+B→C
     │                    output: {data: [{A,B,C}, ...], rows}
     ▼
[MySQL 输出控件]          config: {table: "result"}
     │                    INSERT INTO result (A,B,C) VALUES (...)
     │                    output: {rows}
     ▼
Execution 记录写入:  {status:"success", rows_processed: N}
```

## 7. 接口设计


### 7.2 控件注册接口

每个控件通过 Python 类的属性暴露其元信息，前端通过 GET /api/components/built-in 获取。前端根据 config_schema 动态渲染配置表单，无需为每种控件硬编码表单组件。

```json
// GET /api/components/built-in 响应示例
[
  {
    "name": "csv-input",
    "display_name": "CSV 输入",
    "type": "input",
    "config_schema": {
      "type": "object",
      "properties": {
        "path": {"type": "string", "title": "文件路径"},
        "separator": {"type": "string", "title": "分隔符", "default": ","}
      },
      "required": ["path"]
    }
  },
  ...
]
```
